# Этап 1. Foundation-архитектура

Последнее обновление: 2026-03-16

## 1. Цель этапа

Этап 1 фиксирует решения, которые должны быть приняты до начала scaffold-реализации:

- границы первой версии;
- модель `local-first`;
- общую структуру Flutter-проекта;
- состав доменных модулей;
- экранную карту;
- контракты локального хранения и файловой синхронизации;
- порядок реализации этапа 2.

Этап 1 не включает создание Flutter-проекта, генерацию платформенных папок, миграции или executable-кода.

## 2. Границы v1

В первую версию входят:

- одно Flutter-приложение с поддержкой Windows и Android;
- локальная SQLite-база на каждом устройстве;
- локальное хранение изображений компонентов, подписей и PDF;
- Windows как мастер-узел администрирования;
- Android как клиент прохождения проверок;
- пакетная синхронизация через Яндекс.Диск;
- журнал действий, корзина и резервное копирование на Windows.

В первую версию не входят:

- серверная часть;
- Supabase и любые платные backend-сервисы;
- веб-админка;
- push-уведомления;
- LDAP/SSO;
- совместное онлайн-редактирование;
- редактирование завершенных проверок;
- автоматическое слияние конфликтующих результатов.

## 3. Обязательные инварианты

- Windows редактирует справочники, Android не редактирует справочники.
- Android может создавать только проверки, результаты, подписи и локальные PDF.
- Завершенная проверка immutable, разрешены только чтение и передача на синхронизацию.
- Пакет справочников публикуется только Windows.
- Пакет результата проверки публикуется только Android.
- При наличии сети Android должен проверять lock перед стартом проверки.
- При отсутствии сети проверка допускается, а конфликт проверяется только при последующей синхронизации.

## 4. Структура будущего Flutter-проекта

Целевая структура репозитория:

```text
/
  lib/
    app/
    core/
    data/
      sqlite/
      storage/
      sync/
    features/
      auth/
      catalog/
      objects/
      components/
      checklists/
      inspections/
      sync/
      audit/
      settings/
    ui/
  test/
  integration_test/
  assets/
    icons/
    seed/
```

Назначение уровней:

- `app/` содержит точку входа, конфигурацию маршрутов и composition root;
- `core/` содержит общие типы, ошибки, конфигурацию, время, идентификаторы и платформенные абстракции;
- `data/sqlite/` содержит открытие БД, схемы, миграции и репозитории;
- `data/storage/` содержит доступ к локальным файлам, путям и резервным копиям;
- `data/sync/` содержит упаковку, распаковку и transport-адаптер Яндекс.Диска;
- `features/*` содержат доменную логику и экранные потоки;
- `ui/` содержит общие виджеты и тему.

## 5. Модульная декомпозиция

### `auth`

- выбор пользователя;
- проверка PIN;
- хранение активной локальной сессии;
- матрица ролей.

### `catalog`

- предприятия, цеха, участки;
- пользователи и роли;
- справочники верхнего уровня.

### `objects`

- изделия;
- иерархия объектов проверки;
- дерево состава изделия.

### `components`

- компоненты объекта;
- галерея изображений;
- локальные пути и метаданные медиа.

### `checklists`

- чек-листы;
- пункты чек-листа;
- привязки чек-листов к объектам и компонентам.

### `inspections`

- черновики и активные проверки;
- результаты по пунктам;
- подписи;
- локальная генерация PDF;
- история проверок.

### `sync`

- публикация пакета справочников;
- прием пакета результата;
- очередь исходящих и входящих пакетов;
- диагностика;
- lock-файлы и контроль конфликтов.

### `audit`

- журнал действий;
- события sync;
- конфликты;
- системные ошибки.

### `settings`

- конфигурация устройства;
- токен Яндекс.Диска;
- пути локального хранения;
- параметры резервного копирования.

## 6. Локальное файловое хранилище

Базовый корень:

```text
/app_data/
  /db/
    main.db
  /media/
    /components/
      /{component_id}/
    /signatures/
      /{inspection_id}/
    /reports/
      /{inspection_id}/
  /sync/
    /incoming/
    /outgoing/
    /processed/
    /temp/
  /cache/
    /images/
  /logs/
```

Правила:

- все пути в БД хранятся относительными к корню `/app_data`;
- компонентные изображения хранятся отдельно от подписей и PDF;
- входящие и исходящие пакеты не смешиваются;
- после успешной обработки пакет перемещается в `/sync/processed`;
- временные файлы архивирования и распаковки пишутся только в `/sync/temp`.

## 7. Структура папок на Яндекс.Диске

```text
/company_qc_app/
  /manifest/
    global_manifest.json
  /reference_data/
    /snapshots/
    /packages/
  /devices/
    /{device_id}/
      device_manifest.json
  /results/
    /incoming/
    /processed/
    /conflicts/
  /media/
    /components/
  /locks/
  /backup/
  /trash/
```

Правила:

- каталог `reference_data/packages` содержит опубликованные Windows пакеты справочников;
- каталог `results/incoming` содержит еще не обработанные result-пакеты Android;
- каталог `results/processed` содержит уже импортированные result-пакеты;
- каталог `results/conflicts` хранит конфликтные пакеты без изменения содержимого;
- каталог `media/components` используется только для мастер-медиа компонентов;
- PDF и подписи результата идут внутри result-пакета, а не отдельными upload-операциями;
- `device_manifest.json` хранит диагностическое состояние устройства, но не является источником истины для бизнес-данных.

## 8. Роли и ограничения доступа

### `administrator`

- управляет справочниками;
- публикует reference-пакеты;
- импортирует result-пакеты;
- работает с журналом, корзиной и резервным копированием.

### `worker`

- запускает проверку;
- проходит чек-лист;
- завершает проверку;
- ставит результат в очередь на отправку.

### `commission`

- участвует в проверке;
- подписывает результат;
- просматривает историю проверок.

### `viewer`

- имеет доступ только на чтение к доступным данным и результатам.

Ограничения:

- редактирование справочников разрешено только роли `administrator` и только в Windows;
- Android не получает экранов редактирования справочников;
- завершенная проверка недоступна для изменения на любой роли.

## 10. Порядок реализации этапа 2

Этап 2 должен идти в следующем порядке:

1. создать Flutter-проект и включить Windows и Android;
2. создать каркас директорий `lib/app`, `lib/core`, `lib/data`, `lib/features`, `test`, `integration_test`, `assets`;
3. подключить SQLite, файловое хранилище и безопасное хранение секрета;
4. реализовать открытие локальной БД и версию схемы;
5. реализовать резолвер локальных путей `/db`, `/media`, `/sync`, `/cache`, `/logs`;
6. реализовать базовый журнал событий приложения;
7. подготовить пустые shell-экраны Windows и Android с проверкой ролей и маршрутизации;
8. только после этого переходить к мастер-данным Windows и inspection flow Android.

## 11. Артефакты этапа 1

Этап 1 считается завершенным, когда в репозитории зафиксированы:

- журнал архитектурных решений;
- foundation-архитектура проекта;
- контракт SQLite и sync-пакетов;
- обновленный чек-лист с отмеченными проектировочными пунктами.

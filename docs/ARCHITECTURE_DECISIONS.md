# Журнал архитектурных решений

Последнее обновление: 2026-03-16

## ADR-001. Границы первой версии

Статус: утверждено

Решение:

- первая версия включает Windows-приложение администратора, Android-приложение проверки, локальное SQLite-хранилище, локальные медиафайлы и пакетную синхронизацию через Яндекс.Диск;
- первая версия не включает серверную часть, платные backend-сервисы, push-уведомления, веб-админку, LDAP/SSO и онлайн-совместное редактирование;
- завершенные проверки неизменяемы;
- автоматическое разрешение сложных конфликтов не поддерживается.

Причина:

- эти ограничения уже зафиксированы в ТЗ и снижают архитектурный риск на старте;
- объем первой версии остается совместимым с моделью `local-first`.

## ADR-002. Источник истины и владение данными

Статус: утверждено

Решение:

- Windows-приложение является единственным источником истины для справочников, структуры предприятия, структуры изделий, компонентов, чек-листов и пользователей;
- Android-приложение получает только опубликованные справочные данные и создает только результаты проверок;
- поток справочников однонаправленный: `Windows -> Android`;
- поток результатов проверок однонаправленный: `Android -> Windows`.

Причина:

- это минимизирует конфликтность данных;
- это упрощает проверку целостности sync-пакетов и снимает необходимость двустороннего merge для мастер-данных.

## ADR-003. Архитектурный каркас Flutter-проекта

Статус: утверждено

Решение:

- используется одна кодовая база Flutter для Windows и Android;
- код организуется по feature/layer структуре;
- platform-specific поведение изолируется в адаптерах, а не размазывается по feature-логике;
- базовые подсистемы проекта: `auth`, `catalog`, `objects`, `components`, `checklists`, `inspections`, `sync`, `audit`, `settings`, `storage`.

Причина:

- общая модель данных, синхронизация и offline-поведение должны переиспользоваться между платформами;
- раздельные кодовые базы на старте увеличат стоимость изменений в модели и sync-контрактах.

## ADR-004. Контракт локального хранения

Статус: утверждено

Решение:

- каждое устройство хранит SQLite-базу, медиа, PDF, sync-очередь и журналы в локальной рабочей папке приложения;
- локальное файловое хранилище имеет фиксированные каталоги `/db`, `/media`, `/sync`, `/cache`, `/logs`;
- в базе хранятся ссылки на локальные файлы и служебные метаданные, а не бинарное содержимое файлов.

Причина:

- это упрощает резервное копирование и миграции схемы;
- медиафайлы и PDF не раздувают SQLite и легче восстанавливаются из архива.

## ADR-005. Контракт синхронизации

Статус: утверждено

Решение:

- Яндекс.Диск используется только как файловый транспорт;
- синхронизация выполняется пакетами, а не CRUD-операциями по отдельным сущностям;
- справочники публикуются Windows в виде версионированного пакета;
- результат проверки публикуется Android в виде отдельного zip-пакета с JSON и вложенными файлами;
- конфликтующие результаты не объединяются автоматически.

Причина:

- пакетная модель лучше соответствует `local-first`;
- transport-only подход не делает Яндекс.Диск удаленной базой данных.

## ADR-006. Версионирование и идентификаторы

Статус: утверждено

Решение:

- все синхронизируемые сущности используют `id` в формате UUID;
- для синхронизируемых записей обязательны поля `version`, `created_at`, `updated_at`, `deleted_at`, `is_deleted`;
- для пакетов обязательны поля `package_id`, `package_type`, `schema_version`, `created_at`, `checksum`;
- версия схемы приложения и версия sync-схемы ведутся независимо.

Причина:

- это позволяет независимо эволюционировать локальную БД и формат обмена;
- единый контракт версионирования нужен для repeatable sync и диагностики.

## ADR-007. Стратегия этапов после проектирования

Статус: утверждено

Решение:

- этап 1 закрывает архитектуру, контракты хранения, sync и scope первой версии;
- этап 2 создает Flutter-каркас, базовую структуру модулей, открытие SQLite, файловое хранилище и журналирование;
- этап 3 реализует Windows master-data flows;
- этап 4 реализует Android inspection flows;
- этап 5 реализует обмен пакетами через Яндекс.Диск;
- этап 6 закрывает тестирование и эксплуатационные сценарии.

Причина:

- сначала нужно стабилизировать модель данных и sync-контракты;
- после этого реализация может идти без повторного выбора архитектурных решений.

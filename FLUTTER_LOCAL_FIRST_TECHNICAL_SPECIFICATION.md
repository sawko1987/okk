# Техническое задание

## 1. Название проекта

Кроссплатформенное приложение контроля комплектности и приемки изделий для:

- Windows-приложения администратора;
- Android-приложения для сотрудников и представителей комиссии.

## 2. Цель проекта

Разработать полностью бесплатное приложение без использования платных облачных сервисов, которое:

- работает офлайн;
- синхронизируется между устройствами через Яндекс.Диск;
- хранит данные локально;
- позволяет администратору управлять изделиями, структурой и чек-листами;
- позволяет сотрудникам и представителям комиссии проводить проверки на Android-устройствах;
- сохраняет результаты проверок, подписи, изображения и PDF-отчеты.

## 3. Базовая концепция

Приложение строится по модели `local-first`.

Это означает:

- каждое устройство хранит свои рабочие данные локально;
- основная база справочников находится в Windows-приложении администратора;
- Android-устройства получают только нужные данные для работы;
- результаты проверок создаются локально на телефоне;
- синхронизация выполняется пакетно через Яндекс.Диск;
- Яндекс.Диск используется как транспорт файлов синхронизации, а не как онлайн-база данных.

## 4. Платформы

Необходимо разработать:

- Flutter-приложение для Windows;
- Flutter-приложение для Android.

Windows-версия должна поддерживать portable-формат запуска без обязательной установки через Store.

## 5. Основные требования

### 5.1 Функциональные требования

Система должна обеспечивать:

- ведение структуры предприятия;
- ведение справочника изделий;
- ведение структуры изделия;
- ведение чек-листов;
- загрузку нескольких изображений для каждого компонента;
- хранение подписей как изображений;
- проведение проверок на Android;
- формирование PDF-отчета;
- просмотр истории проверок;
- ведение журнала действий;
- корзину удаленных данных;
- синхронизацию через Яндекс.Диск;
- ручную синхронизацию по кнопке;
- автоматическую синхронизацию при запуске приложения;
- хранение данных не менее 3 лет.

### 5.2 Нефункциональные требования

Система должна:

- работать без постоянного интернета;
- быть полностью бесплатной в эксплуатации;
- не использовать Supabase и любые платные backend-сервисы;
- обеспечивать устойчивую работу при временном отсутствии сети;
- иметь предсказуемую модель синхронизации;
- исключать редактирование завершенных проверок;
- минимизировать риск конфликтов данных;
- хранить данные локально в SQLite;
- хранить медиафайлы локально на устройстве.

## 6. Пользователи и роли

### 6.1 Роли

В системе используются роли:

- администратор;
- работник;
- представитель комиссии;
- режим просмотра.

### 6.2 Права ролей

#### Администратор

Имеет право:

- создавать и редактировать структуру предприятия;
- создавать и редактировать изделия;
- создавать и редактировать состав изделия;
- создавать и редактировать чек-листы;
- загружать изображения;
- создавать пользователей;
- просматривать все результаты;
- выполнять синхронизацию;
- импортировать и экспортировать данные;
- восстанавливать данные из корзины;
- просматривать журнал действий.

#### Работник

Имеет право:

- выбирать доступное изделие;
- загружать локально данные для проверки;
- проходить проверку;
- формировать результат;
- отправлять результат на синхронизацию;
- просматривать список проверок.

#### Представитель комиссии

Имеет право:

- участвовать в проверке;
- подписывать результат как изображение;
- просматривать историю проверок;
- работать в режиме приемки.

#### Режим просмотра

Имеет право:

- только просматривать данные и результаты без редактирования.

## 8. Архитектура решения

## 8.1 Общая архитектура

Система должна состоять из:

- Windows-приложения администратора;
- Android-приложения проверки;
- локальной базы SQLite на каждом устройстве;
- локального файлового хранилища на каждом устройстве;
- синхронизационного слоя через Яндекс.Диск.

## 8.2 Основной принцип синхронизации

Синхронизация должна быть пакетной, а не实时-запросами к облаку.

### Поток данных

#### Из Windows в Android

Передаются:

- пользователи;
- структура предприятия;
- изделия;
- состав изделий;
- чек-листы;
- изображения компонентов;
- метаданные;
- версии данных.

#### Из Android в Windows

Передаются:

- завершенные проверки;
- подписи;
- PDF-отчеты;
- журналы действий по проверкам;
- служебные метаданные устройства.

## 8.3 Источник истины

Источником истины для справочников является Windows-приложение.

Android-устройства:

- не редактируют справочники;
- не редактируют структуру;
- не редактируют чек-листы;
- создают только результаты проверок.

Это решение снижает вероятность конфликтов и повышает надежность.

## 9.1 База данных

В базе должны храниться:

- пользователи;
- чек-листы;
- проверки;
- результаты;
- ссылки на локальные изображения;
- версии данных;

- журнал действий;

## 9.2 Локальное файловое хранилище

```text
    /components/
    /signatures/
    /reports/
  /sync/
    /incoming/
    /outgoing/
    /temp/

  /logs/
```

## 10. Использование Яндекс.Диска

Яндекс.Диск используется только как транспорт обмена.

На Яндекс.Диске не выполняются:

- SQL-запросы;
- прямые CRUD-операции по сущностям;
- онлайн-блокировки на постоянной основе;
- работа как с удаленной базой данных.

На Яндекс.Диске хранятся:

- пакеты синхронизации;
- манифесты версий;
- изображения;
- PDF;
- резервные копии;
- lock-файлы;
- архивы;
- корзина синхронизации при необходимости.

## 11. Модель синхронизации

## 11.1 Тип синхронизации

Поддерживаются режимы:

- автоматическая при запуске;
- ручная по кнопке `Синхронизировать сейчас`;
- синхронизация перед началом проверки;
- синхронизация после завершения проверки;
- фоновая попытка синхронизации при появлении сети.

## 11.2 Направления синхронизации

### Справочники

Только:

- Windows -> Android

### Результаты проверок

Только:

- Android -> Windows

## 11.3 Пакет синхронизации справочников

Пакет справочников должен содержать:

- версию пакета;
- дату формирования;
- устройство-источник;
- список объектов;
- список чек-листов;
- список пользователей;
- список изменений;
- ссылки на медиафайлы;
- список обязательных файлов для скачивания.

## 11.4 Пакет результата проверки

Пакет результата должен содержать:

- id проверки;
- id устройства;
- id пользователя;
- id изделия;
- тип объекта;
- дату и время начала;
- дату и время завершения;
- статус проверки;
- результаты по пунктам;
- подписи;
- путь к PDF;
- контрольную сумму пакета;
- версию схемы.

## 11.5 Версионирование

Для всех синхронизируемых сущностей необходимо использовать:

- `entity_id`;
- `version`;
- `updated_at`;
- `deleted_at`;
- `is_deleted`.

Для пакетов:

- `package_id`;
- `package_type`;
- `created_at`;
- `checksum`;
- `schema_version`.

## 12. Правила конфликтов

## 12.1 Основная стратегия

Конфликты должны быть максимально исключены архитектурно.

Решения:

- справочники изменяются только в Windows-приложении;
- Android не может редактировать справочники;
- завершенная проверка не может редактироваться;
- одна активная проверка на одно изделие на одном устройстве;
- при наличии сети приложение должно проверять актуальность перед стартом проверки.

## 12.2 Поведение при конфликте

Конфликтом считается ситуация, когда:

- одно и то же изделие уже имеет активную проверку на другом устройстве;
- устройство пытается отправить результат по устаревшей версии объекта;
- в систему приходит дублирующий пакет результата.

В этом случае:

- новая проверка не стартует, если удалось проверить lock и обнаружен конфликт;
- уже начатая офлайн-проверка может быть завершена;
- при синхронизации конфликт помечается в журнале;
- администратор получает уведомление;
- автоматическое объединение конфликтующих проверок не выполняется.

## 12.3 Lock-механизм

При наличии сети перед стартом проверки приложение должно:

- проверить наличие lock-файла по изделию;
- если lock-файла нет, создать lock-файл;
- записать в lock:
  - id устройства;
  - id пользователя;
  - id изделия;
  - время создания;
  - время истечения lock;
- после завершения проверки удалить lock-файл.

Если сети нет:

- lock не создается;
- проверка допускается на локально загруженных данных;
- при синхронизации выполняется проверка на конфликт.

## 13. Правила офлайн-работы

Приложение Android должно:

- запускаться без интернета;
- открывать ранее синхронизированные данные;
- отображать локально загруженные изображения;
- позволять проходить проверку полностью офлайн;
- формировать PDF офлайн;
- ставить результаты в очередь на отправку.

Если интернет отсутствует:

- работа не блокируется;
- справочники не обновляются;
- результаты хранятся локально до следующей синхронизации.

## 19. Изображения и медиа

## 19.1 Изображения компонентов

Для каждого компонента должно поддерживаться несколько изображений.

Требования:

- изображения загружает администратор;
- изображения синхронизируются на Android только для выбранных данных;
- изображения должны отображаться без сети;
- изображения должны кешироваться локально;
- в базе должны храниться метаданные и локальные пути.

## 19.2 Подписи

Подпись хранится как изображение.

Требования:

- подпись создается в приложении;
- подпись сохраняется локально;
- подпись прикрепляется к результату проверки;
- подпись синхронизируется на Windows;
- подпись отображается в PDF.

## 19.3 PDF

PDF-отчет формируется локально на Android.

Требования:

- PDF должен содержать:
  - данные изделия;
  - состав проверки;
  - результаты;
  - дату и время;
  - подписи;
  - участников проверки;
- PDF сохраняется локально;
- PDF отправляется на Windows через синхронизацию;
- PDF может быть отправлен с телефона по почте или через мессенджер.

## 20. Хранение и архивирование

## 20.1 Срок хранения

Данные должны храниться не менее 3 лет.

## 20.2 Архив

Старые данные могут быть перенесены в архив без удаления из резервной копии.

Минимально требуется:

- хранить все записи 3 года;
- поддерживать резервный экспорт;
- не удалять результаты автоматически раньше срока.

## 20.3 Корзина

Система должна поддерживать корзину.

В корзину помещаются:

- удаленные изделия;
- удаленные компоненты;
- удаленные чек-листы;
- удаленные пользователи;
- удаленные медиа.

Корзина должна позволять:

- просмотр;
- восстановление;
- окончательное удаление.

## 21. Журнал действий

Система должна вести журнал действий.

В журнал записываются:

- вход пользователя;
- создание сущности;

- id события;
- дата и время;
- пользователь;
- устройство;
- тип действия;

С учетом требования простоты рекомендуется:

## 26. Производительность

Система должна:

- быстро открывать локальные данные;
- загружать только нужные цеха или изделия;
- не скачивать лишние изображения;
- использовать контрольные суммы для проверки изменений;
- использовать локальный кеш изображений;
- работать на 3 Android-устройствах и 1 Windows-устройстве без постоянного сервера.

## 27. Диагностика

Приложение должно содержать экран диагностики, где отображаются:

- версия приложения;
- версия локальной схемы БД;
- id устройства;
- время последней синхронизации;
- статус подключения к Яндекс.Диску;
- количество пакетов в очереди;
- ошибки последней синхронизации;
- количество локальных данных;
- путь к локальной базе;
- свободное место.

## 28. Ограничения и принятые решения

В рамках данного проекта фиксируются следующие решения:

- Supabase полностью исключается;
- проект создается заново;
- основной стек: Flutter + SQLite;
- админка переносится из веб-формата в Windows-приложение;
- Android-приложение предназначено только для прохождения проверки;
- редактирование справочников с Android запрещено;
- завершенные проверки неизменяемы;
- изображения загружает только администратор;
- синхронизация идет через Яндекс.Диск;
- система должна работать без постоянного интернета.

## 29. Что не входит в первую версию

В первую версию не входят:

- серверная часть;
- платные облачные сервисы;
- push-уведомления;
- онлайн-совместное редактирование;
- сложная ролевая модель с LDAP/SSO;
- веб-админка;
- редактирование завершенных проверок;
- автоматическое разрешение сложных конфликтов;
- печать напрямую из мобильного приложения.

## 31. Критерии приемки

Система считается принятой, если:

- Windows-приложение работает локально без внешнего backend;
- Android-приложение работает офлайн;
- синхронизация через Яндекс.Диск работает;
- изображения компонентов отображаются во время проверки;
- несколько изображений на компонент поддерживаются;
- подписи сохраняются как изображения;
- PDF формируется корректно;
- завершенные проверки не редактируются;
- журнал действий ведется;
- корзина работает;
- резервная копия создается и восстанавливается;
- синхронизация выполняется между 1 Windows и 3 Android-устройствами;
- данные справочников идут с Windows на Android;
- результаты идут с Android на Windows;
- Supabase полностью отсутствует в новой системе.
